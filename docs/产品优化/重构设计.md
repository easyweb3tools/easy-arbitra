# Easy Arbitra 产品重构设计方案

更新时间：2026-02-24
视角：产品经理（用户体验 + 转化导向）
输入依据：`docs/运营优化.md` + 现有代码库实际能力

---

## 1. 现状诊断

### 1.1 当前产品架构

| 页面 | 核心能力 | 用户价值 |
|------|---------|---------|
| 首页 `/` | 索引统计、新增钱包、Top 表现 | 概览，但无行动引导 |
| 钱包列表 `/wallets` | 筛选、分页、关注切换 | 浏览，但标签同质化严重 |
| 钱包详情 `/wallets/[id]` | 三层归因、AI 报告、分享卡 | 信息丰富，但缺"跟不跟"的结论 |
| 排行榜 `/leaderboard` | 评分排序、策略分类筛选 | 有排名，但"为什么排前面"不直观 |
| 异常流 `/anomalies` | 异常检测、严重度筛选 | 专业，但与跟单决策割裂 |
| Watchlist `/watchlist` | 关注列表、事件流 | 有更新，但缺少可执行建议 |
| 方法论 `/methodology` | 三层框架解释 | 教育性强，但用户不会主动看 |

### 1.2 核心问题映射

运营优化文档指出四个关键问题，对应到产品设计层面：

| 运营问题 | 产品根因 | 影响面 |
|---------|---------|--------|
| 标签同质化 | 钱包卡片信息架构扁平，缺乏视觉分层 | 首页、列表、排行榜 |
| AI 摘要覆盖不足 | 空状态体验差，无降级方案 | 全局信任感 |
| 跟单闭环不完整 | 详情页是"信息面板"而非"决策面板" | 转化率 |
| 传播位价值未释放 | 分享卡是附属功能，非一等公民 | 增长 |

## 2. 重构目标

将产品从 **"分析工具"** 重构为 **"跟单决策引擎"**。

用户心智模型转变：

```
当前：我来看看有哪些厉害的钱包 → 好多数据 → 然后呢？
目标：我来找值得跟的钱包 → 这个适合我 → 一键关注 → 有事提醒我
```

核心衡量标准：**用户从进入到完成首次关注的路径步数和耗时。**

## 3. 信息架构重构

### 3.1 页面结构调整

```
重构前                          重构后
────────                       ────────
首页（统计概览）                  首页（发现 + 推荐）
钱包列表（全量浏览）               ↳ 今日精选（3 个明星钱包）
钱包详情（信息展示）               ↳ 策略组合包入口
排行榜（评分排序）                 ↳ 热门变动信号
异常流（专业监控）
Watchlist（关注列表）            探索页（合并列表 + 排行榜）
方法论（教育）                    ↳ 按策略类型分 Tab
                               ↳ 钱包卡片重设计
                               ↳ 内联排序/筛选

                               钱包详情（决策面板）
                               ↳ 跟单决策卡（首屏）
                               ↳ 三层归因（折叠展开）
                               ↳ AI 报告（摘要优先）
                               ↳ 分享卡（行动区）

                               我的跟踪（升级 Watchlist）
                               ↳ 动作流（可执行事件）
                               ↳ 组合概览
                               ↳ 风险雷达

                               异常中心（从导航降级）
                               ↳ 与钱包详情联动
                               ↳ 可执行建议
```

### 3.2 导航结构

```
主导航（4 项，精简自 7 项）
├── 发现        → 首页推荐 + 探索
├── 探索        → 合并后的钱包列表/排行榜
├── 我的跟踪    → 升级版 Watchlist
└── 更多        → 异常中心 / 方法论 / 设置
```

**设计原则**：导航项 ≤ 5 个，用户认知负荷最小化。

## 4. 核心页面重设计

### 4.1 首页重设计：从"仪表盘"到"发现引擎"

**当前问题**：首页展示索引统计（多少市场、多少钱包），对用户决策无直接帮助。

│  │ 稳健型    │ │ 事件驱动  │ │ 做市策略  │     │
│  │ 7D: +12% │ │ 7D: +28% │ │ 7D: +8%  │     │
│  │ 风险: 低  │ │ 风险: 中  │ │ 风险: 低  │     │
│  │ ⭐ 关注   │ │ ⭐ 关注   │ │ ⭐ 关注   │     │
│  └──────────┘ └──────────┘ └──────────┘     │
├──────────────────────────────────────────────┤
│  新手组合包                                   │
│  ┌─────────────────┐ ┌─────────────────┐    │

│  │ 最大回撤: <10%    │ │ 最大回撤: <30%  │    │

**关键设计决策**：

1. **今日精选**：来自运营文档的"明星池"概念，每日 3 个，由后端 `ops/highlights` 接口驱动，增加"为什么推荐"一句话。
2. **新手组合包**：新增概念，降低新用户首次关注门槛——不用自己选，直接选一个组合。
3. **实时信号**：整合 anomaly feed + watchlist events，展示全局有价值的变动。

**需要的后端改动**：

| 改动 | 涉及文件 | 说明 |
|------|---------|------|
| 新增钱包池分层字段 | `model/models.go` → `WalletScore` | 增加 `pool_tier` (观察/策略/明星) |
| 新增组合包 API | `handler/handlers.go`, `router.go` | `GET /api/v1/portfolios` 返回预定义组合 |
| 扩展 highlights API | `handler/handlers.go` | 返回推荐理由文案 |
| 批量关注 API | `handler/handlers.go` | `POST /api/v1/watchlist/batch` |

### 4.2 探索页重设计：合并列表与排行榜

**当前问题**：`/wallets` 和 `/leaderboard` 功能重叠，用户需要在两个页面间切换对比。

**重设计方案**：合并为统一的探索页，用 Tab 区分浏览模式。

```
┌──────────────────────────────────────────────┐
│  探索钱包                                     │
│  ┌────┐ ┌────┐ ┌────────┐ ┌────────┐        │
│  │ 全部 │ │ 做市 │ │ 事件驱动 │ │ 量化分析 │ ...   │
│  └────┘ └────┘ └────────┘ └────────┘        │
│                                              │
│  排序: [评分▼] [7D收益] [稳定性] [信息优势]     │
│  筛选: [风险等级] [池等级] [有AI报告]           │
│                                              │
│  ┌──────────────────────────────────────┐    │
│  │ #1  0x1a..              明星池 · 稳健 │    │
│  │     做市策略 · 信息优势: 中等              │    │
│  │     ┌──────────────────────────┐     │    │
│  │     │ 7D: +12%  30D: +35%     │     │    │
│  │     │ 回撤: -3%  稳定性: 92    │     │    │
│  │     └──────────────────────────┘     │    │
│  │     "持续通过做市赚取 rebate，回撤极小"   │    │
│  │                        [关注] [详情]  │    │
│  ├──────────────────────────────────────┤    │
│  │ #2  0x2b..              策略池 · 进攻 │    │
│  │     ...                              │    │
│  └──────────────────────────────────────┘    │
└──────────────────────────────────────────────┘
```

**关键改进**：

1. **策略类型作为一级 Tab**：利用 Layer 2 已有的 6 种分类（Market Maker / Event Trader / Quant / Arbitrage / Noise / Lucky），让用户按自己偏好快速聚焦。
2. **卡片信息分层**：
   - 第一层（扫一眼）：排名 + 地址 + 池等级标签 + 风格标签
   - 第二层（停下来看）：核心数字（收益、回撤、稳定性）
   - 第三层（做决定）：一句话 AI 摘要 + 操作按钮
3. **池等级徽章**：明星池（金色）/ 策略池（银色）/ 观察池（灰色），视觉上立即区分。
4. **"有 AI 报告"筛选**：解决"No AI summary yet"问题——不是隐藏，而是让用户主动选择只看有完整报告的钱包。

**前端改动**：

| 改动 | 涉及文件 | 说明 |
|------|---------|------|
| 合并页面路由 | `app/wallets/page.tsx` | 吸收 leaderboard 的排序逻辑 |
| 重构钱包卡片组件 | 新建 `components/WalletCard.tsx` | 统一的分层卡片 |
| Tab 筛选组件 | 新建 `components/StrategyTabs.tsx` | 按策略类型切换 |
| 移除 leaderboard 路由 | `app/leaderboard/` | 重定向到 `/wallets?sort=score` |

│  │                                        │  │

### 4.4 Watchlist 升级："我的跟踪"

**当前问题**：Watchlist 是关注列表 + 事件流，但事件缺乏可执行性，用户看到更新不知道该做什么。

**重设计方案**：

```
┌──────────────────────────────────────────────┐
│  我的跟踪                                     │
│                                              │
│  ┌─── 组合概览 ────────────────────────────┐  │
│  │ 关注中: 5 个钱包                         │  │
│  │ 风格分布: 稳健×2  进攻×2  事件×1          │  │
│  │ 整体状态: 🟢 4/5 正常  🟡 1 需关注        │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  ┌─── 需要行动 ────────────────────────────┐  │
│  │ 🔴 0x4d.. 异常回撤 -12%                  │  │
│  │    建议: 考虑暂停跟踪，等待企稳            │  │
│  │                       [查看详情] [取消关注] │  │
│  ├────────────────────────────────────────┤  │
│  │ 🟡 0x2b.. 策略漂移：从做市转向事件驱动      │  │
│  │    建议: 重新评估是否符合你的风格           │  │
│  │                       [查看详情] [保持关注] │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  ┌─── 正常更新 ────────────────────────────┐  │
│  │ 📊 0x1a.. 新 AI 报告 · 2h前              │  │
│  │ ── 0x3c.. 日常交易 3 笔 · 5h前           │  │
│  │ 📊 0x5e.. 新 AI 报告 · 8h前              │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

**关键改进**：

1. **组合概览**：一眼看清关注组合的整体状态，不用逐个钱包查看。
2. **事件分级**：把事件分为"需要行动"和"正常更新"两个区域——用户只需关注红色和黄色。
3. **每条事件带建议**：不只是"发生了什么"，而是"发生了什么 + 建议做什么"。
4. **内联操作**：直接在事件流中提供操作按钮，减少页面跳转。

**需要的后端改动**：

| 改动 | 涉及文件 | 说明 |
|------|---------|------|
| 事件分级逻辑 | `service/watchlist_service.go` | 事件增加 `action_required` 和 `suggestion` 字段 |
| 组合概览 API | `handler/handlers.go` | `GET /api/v1/watchlist/summary` |
| 扩展 feed API | `handler/handlers.go` | feed 返回增加建议文案 |

## 5. 钱包池分层系统设计

运营文档提出三层钱包池（观察 / 策略 / 明星），需要落地为产品功能。

### 5.1 分层规则

| 池等级 | 入池条件 | 展示策略 | 淘汰条件 |
|-------|---------|---------|---------|
| 观察池 | 被系统识别为 potential wallet | 不出现在推荐位，仅在探索页可筛选 | 30 天无显著表现 → 移除追踪 |
| 策略池 | 连续 7D 正收益 + 回撤 <15% + AI 报告完整 | 出现在探索页，有银色标签 | 回撤 >20% 或 14D 无交易 → 降级观察池 |
| 明星池 | 连续 30D 表现稳定 + 回撤 <10% + 策略可解释 | 首页推荐位，金色标签，优先生成 AI 报告 | 回撤 >15% 或策略漂移 → 降级策略池 |

### 5.2 自动升降级

```
Worker: pool_tier_evaluator (每日运行)
  ├── 读取 wallet_features_daily + wallet_score
  ├── 计算连续表现指标
  ├── 判定升降级
  ├── 更新 wallet_score.pool_tier
  └── 生成 wallet_update_event (池变动事件)
```

**涉及文件**：
- 新增 `worker/pool_evaluator.go`
- 修改 `model/models.go`：`WalletScore` 增加 `PoolTier string` 字段
- 新增 migration：`005_pool_tier.sql`

## 6. 空状态与降级策略

解决 "No AI summary yet" 问题不能只靠提高覆盖率，还需要产品层面的降级方案。

### 6.1 分级降级策略

| 场景 | 当前体验 | 重设计后 |
|------|---------|---------|
| 无 AI 报告 | "No AI summary yet" | 展示基于规则的自动摘要（Layer 1-2 数据生成） |
| AI 报告过期（>7D） | 无提示 | 标记"报告生成于 X 天前"+ 提供刷新按钮 |
| 无足够交易数据 | 照常展示 | 标记"数据样本不足，结论可靠性较低" |
| 钱包不活跃（>14D） | 照常展示 | 标记"该钱包已 X 天未交易" |

### 6.2 自动摘要生成规则

当 AI 报告不可用时，基于已有的 `WalletScore` 和 `WalletFeaturesDaily` 生成模板化摘要：

```
模板：该钱包采用{strategy_type}策略，近 7 天收益 {pnl_7d}%，
      Smart Score {smart_score}。{risk_statement}。

risk_statement 映射：
  score >= 80 → "整体表现稳定"
  score >= 60 → "表现尚可，需持续观察"
  score < 60  → "近期波动较大，建议谨慎"
```

**涉及文件**：
- 新增前端工具函数 `lib/fallback-summary.ts`
- 修改钱包卡片组件，优先展示 AI 摘要，降级展示模板摘要

## 13. 风险控制

1. **合规风险**：所有决策卡内容必须附带免责声明，措辞使用"概率判断"而非"推荐"。
2. **推荐偏差**：明星池钱包需有回溯验证机制，避免推荐后表现大幅偏离。
3. **信息过载**：重构过程中注意信息密度——宁可隐藏，不可堆砌。
4. **技术债务**：合并列表和排行榜时做好旧路由重定向，不丢失已有 UTM 归因数据。
5. **渐进式发布**：每个 Phase 独立可上线，不形成"全部做完才能用"的依赖。
